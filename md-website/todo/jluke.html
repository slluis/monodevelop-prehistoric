John's MonoDevelop Todo List

Here is a list of stuff that is on my todo list for MD. Some of these are huge, others are smaller. All of them need to get done and I will be implementing them over time. They are not listed in any particular order at all. And if anyone starts to implement one and wants my help, or just wants to do it on their own, and submits patch(es) for them I won't be offended. I promise. If you have any questions regarding this list, email either the list or me directly.

Build system

Status: planning

Currently MonoDevelop projects can be built from within the IDE, and there is preliminary support for generating Makefiles. If Makefile generation is permanently decided upon there are many things to fix about it.

I have prototyped a small managed build tool that can build MonoDevelop projects from the command-line. I believe this to be a better approach then Makefile generation, for three reasons:

  • does not require distribution of generated Makefiles
  • will work on win32
  • does not duplicate information

The last issue is to go through and verify the project format satisfies our needs for the long term. My current opinion is that it does, with two exceptions:

  • Alternative resource names
  • Dependency information

Terminal

Status: mostly done and in svn (Dock interaction bug)

Our current method of running projects via xterm is troublesome at best. Using VTE will improve that, and also allow people to perform simple commands from within the IDE itself. There is currently a weird bug when VTE is combined with the gdldock, hopefully it will be easier to debug when the managed dock is completed.

NUnit integration

Status: very early and in svn (but abandoned by me)

Port either gnunit (nunit-gtk in mono cvs) or SharpDevelop's NUnit addin. I started on this, but lost interest because I never really use NUnit myself. This would be a great thing for someone to start out with, and will be helpful to many people.

Graph of Project classes

It should be possible to generate a simple hierarchal image based on the ClassBrowser, and display that in a View. Requires finding a suitable tool to output a png or svg image. At first, it can just be static, but eventually it should update as your code changes.

Assembly Analayzer

Status: partially ported and in svn

Port this from SharpDevelop. It is basically a automatic code analyzer, that detects many types of different rules for your assemblies.

Start page

Status: not started

We should probably have a nice looking, simple start page. I do not like the current one (that is disabled and half-unported). I would recommend starting from scratch. On the other hand, I don't really miss it either.

Howtos

Status: not started

We need several user level, developer level, translator level, and UI level howtos. These could be both for the website and included in a "How to" submenu under Help.

Profiling

Status: not started

AddIns for profiling would be wonderful.

Startup and command line handling

Reorganize the parsing of command line arguments; hopefully, with Mono.GetOptions. Actually, the whole startup procedure is rather odd and should be more direct and simple. It should be able to support the wacky idea of running 2 or more MD instances and still just work with nautilus (by utilizing .desktop file). Hopefully this can be achieved without the socket stuff. (If you have two open a click on a file in nautilus, which one should it open in?)
